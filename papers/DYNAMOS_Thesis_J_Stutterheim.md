One part of this policy will express **where** the data exchange takes place, and **who** has access to the data.\n\nResearch on this topic has pinpointed common data-sharing patterns, known as archetypes, and has provided suggested architectures to create these data-sharing platforms. The ability to access larger datasets can significantly improve the performance of machine learning models, while in fields like medical science, the availability of more patient data holds immense potential. However, for data sharing to be successful, it must align with the desired level of privacy and security of each participating party [1].\n\nTo create a platform for this, various initiatives around the world work on establishing a Digital Data-sharing Marketplaces (DDM)[2]. One such initiative is the Amsterdam Data Exchange Fieldlab (AMDEX) project [2], which aims to develop a DDM prototype that enables participating organizations to maintain control over their data and enforce access and control through policies. One part of this policy will express **where** the data exchange takes place, and **who** has access to the data.\n\nAn important concept within DDMs is a set of common data-sharing patterns, known as archetypes [1, 3]. These archetypes determine which party provides data, which party does computations on this data, and where the results of that data-sharing exchange go as agreed upon in the policies.\n\nThis thesis introduces DYNAMOS: ‘Dynamically Adaptive Microservice-based OS’. Inspired by the way traditional Operating Systems abstract the complexities of computer architectures into standardized core functions, this research focuses on abstracting different data exchange patterns into a unified set of core data exchange microservices that adhere to agreed-upon data exchange policies. ‘Decentralized’ in DDM is actually an important keyword here, which makes DYNAMOS a *distributed* OS.\n\nIn this thesis, we introduce the generic term ‘microservice chains’ to describe various microservice patterns. A chain is a directed acyclic graph (DAG) of microservices that form a valid data exchange scenario that adheres to policy agreements. These chains need to be executed, for this, we use the generic term ‘jobs’. A user request flows through this chain job deployed to the parties set out by the agreements.\n\nThrough this approach, we aim to achieve the following advantages. Firstly, we strive to establish a self-adaptable system capable of seamlessly incorporating changes in agreements without the need for developer intervention. Secondly, by introducing lightweight data exchange microservices, we eliminate the necessity to redeploy the core system, resulting in a more agile and efficient environment. Lastly, the ‘operating system’ will allow for optimizing extra-functional properties, enabling cost and energy savings based on the system’s environment.\n\n### 1.1 Problem statement\n\nTo create a system like we described in the previous paragraph a list of problems needs to be solved. First, we need a system or language to link microservices to both an archetype and a type of data-sharing request.\n\nSecondly, we need adaptability. Per user, or per request of the same user, different architectures or data-exchange patterns (archetypes) could be selected to run. This requires the routing of data and requests to be highly dynamic. This behavior should be able to adapt based on evolving agreements, user-input or extra-functional properties.\n\nA common approach to achieve self-adaptability is the use of a MAPE-K control loop, which stands for (M)onitor, (A)nalyze, (P)lan, (E)xecute, over a shared (K)nowledge base [4]. Part of this thesis is finding out what information (Knowledge) DYNAMOS needs to generate microservice chains and optimize for extra-functional properties, like energy and costs, and how this can be practically used for implementing self-adaptability.\n\n&lt;page_number&gt;5&lt;/page_number&gt;\n\n---\n\n\n## Page 7\n\nCHAPTER 1. We can also view a computer system as consisting of hardware, software, and data. The operating system provides the means for the proper use of these resources in the operation of the computer system. It performs no useful function by itself but provides an environment within which other programs can do useful work. As described by Silberschatz et al., 2018[6].\n\nIn other words, the OS abstracts away the complexities of the system to allow the users to program or use applications that do something useful. An OS contains the following key attributes et al., 2018[6]:\n\n*   Process Management\n*   Memory Management\n*   File system Management\n*   Device Management\n*   User Interface\n*   Security\n*   Networking\n*   Error Handling\n*   Multi-programming\n*   Logging\n\n### 2.1.1 Distributed Operating systems\n\n“A distributed system is a collection of processors that do not share memory or a clock. Instead, each processor has its local memory, and the processors communicate with one another through various communication lines, such as high-speed buses and the Internet.” [6]\n\nIn a distributed operating system the properties of a ‘regular’ OS, listed above are inherited. An extra set of properties attributed to distributed operating systems is listed in table 2.1.\n\nThus, DYNAMOS should, to a certain extent, be able to provide these OS attributes to the user. Configuration is one of the most basic forms of coordination.” [9] ZooKeeper, is one first generic applications that allow developers to write their own coordination primitives for distributed applications. To manage the complexities of distributed systems, other coordination tools have evolved.\n\nThese tools centralize services for configuration management, providing a consistent repository, and ensuring that each node in the system is in sync with the latest configurations. Beyond configuration, these tools keep a cohesive view of the system’s state, be it active or failed nodes. This overview facilitates tasks such as leader election and distributed locking, optimizing resource management. Furthermore, features like “watches” alert nodes about specific system changes.\n\nAmong the tools available, prominent ones like ZooKeeper¹, etcd², Consul³, and Apache Mesos⁴ stand out, each catering to the specific needs of distributed systems.\n\n## 2.3 Amsterdam Data Exchange Fieldlab\n\nThe following text has been distilled from the AMDEX funding proposal in which the Fieldlab goals are outlined [2]. The ‘Amsterdam Data Exchange Fieldlab’ is done by a consortium of the following entities: Amsterdam Internet Exchange B.V., Dexes B.V., Stichting Amsterdam Economic Board, SURFsara B.V., and the University of Amsterdam. The goal of the consortium is to develop a neutral, generic, independent infrastructure that ensures sovereignty in data exchange initiatives with the working title *Amsterdam Data Exchange* (AMDEX). With AMDEX it should be possible to:\n\n*   On the supply side, offer interested parties reliable and fair data markets and contracts\n*   Empower companies to let third parties use their data without giving a copy of the data\n*   Use this FieldLab to demonstrate a few of the possible workflows\n*   Facilitate data on AMDEX infrastructure\n*   Use this lab to perform tests\n\nStudies by the Dutch economic ministry and the European Commission show that the current infrastructure for safely sharing data is not adequate [2]. Even worse, there are a number of foreign large tech companies that gather and own huge amounts of data, of which the consumers giving the data have no control[2]. For primary operations such as data fetching or storing, HTTP REST is the best overall option thanks to its simplicity. If, on the contrary, the service needs to dispatch lots of messages to other services and trigger heavy processes that don’t need to be awaited, AMQP is a better option [24].\n\n## 2.5.3 Microservice best practices\n\nWang *et al.* [12] performed an exploratory study on microservice best practices, conducting in-depth interviews with microservice experts. The below paragraphs depend for a large part on that study, combined with online sources.\n\n### 2.5.3.1 Logging\n\nLogging is a challenge in microservice architectures. If something happens, it would be near impossible to check individual components for logs and correlate these to actual events [25]. Furthermore, setting up logging and monitoring early is crucial to the success of a microservice platform [12].\n\nWithin the Kubernetes environment, there are several off-the-shelf solutions available. In essence, Kubernetes will catch all output from a pod that is written to STDERR and STDOUT. A stack of existing services can capture that output enhance it, store it centrally, and make it visual and able to query.\n\nCommon solutions are the ELK stack, (Elastic search, Logstash, Kibana) or the combination of Promtail, Loki, and Grafana. However, because the application is unaware of the logging solution, all solutions are easily replaceable and can be switched if requirements demand.\n\n### 2.5.3.2 Distributed tracing and Service Mesh\n\nMetrics and traces are two more key components that enhance the monitorability of both the platform and microservice application. Where logs only show what went wrong according to the application. A trace would actually show the entire path, transit time, and time spent in services, of all requests. Experts express that metrics and tracing must be set up as early in the process as possible, for a successful microservice platform [12].\n\nFirst, the distributed tracing ecosystem is very complex. It includes a dizzying array of projects such as Zipkin, Jaeger, OpenTracing, OpenCensus, OpenTelemetry, and many many others, each with partially overlapping sets of functionality. There are complex matrices of which projects can interoperate with which other projects and in which ways. [26]\n\nThese metrics, allow for identifying bottlenecks and spotting errors that at first seemed unrelated. Furthermore, a trace can be visualized which can help both with debugging and even showing parts of the application architecture.\n\nAgain, there are several solutions for metric collection and tracing. Prometheus is the de facto standard for metrics in Kubernetes and Zipkin and Jaeger are well-known tracing solutions.\n\nThis is where a service mesh comes in as a best practice. A service mesh is a dedicated infrastructure layer built right into a platform [27]. A service mesh can handle metric collection, distributed tracing, ingress encrypted traffic, load balancing, canary deployments, and more. All by injecting a sidecar proxy into every microservice and taking over (HTTP) traffic. The most well-known service meshes for Kubernetes are Istio and Linkerd, and other options include NGINX Service Mesh, Open Service Mesh, and HashiCorp Consul.\n\n## 2.5.4 Sidecar pattern\n\nThe sidecar pattern moves common functionalities into a separate container located close to the primary application. The primary application will have one unified way to talk to the sidecar. Shared libraries, logging, messaging, and configuration are common use cases for the sidecar pattern, which is good according to the single responsibility principle [28]. The sidecar pattern also allows the sidecar and primary application to have separate release cycles to allow development from different teams [28].\n\n&lt;page_number&gt;12&lt;/page_number&gt;\n\n---\n\n\n## Page 14\n\nCHAPTER 2. It can efficiently connect services in and across data centers with pluggable support for load balancing, tracing, health checking, and authentication. It is also applicable in the last mile of distributed computing to connect devices, mobile applications, and browsers to backend services. [29]\n\nFor communicating with a sidecar a lightweight protocol is preferred. An HTTP-based REST method is common, but in most cases, gRPC will outperform REST [30, 31]. The following advantages and disadvantages have been identified for this architecture.\n\n### 3.5.1.1 Advantages\n\n*   Resource efficiency, resources are utilized only when a task is being executed.\n*   Security:\n    *   Limited exposure time, ephemeral jobs run for a short duration, which reduces the window of opportunity for attackers to exploit any potential vulnerabilities.\n    *   Reduced attack surface, as these jobs run for a specific task and then terminate, they don’t keep unnecessary services or open ports, thus reducing the available attack surface.\n    *   Fresh state, with each execution of an ephemeral job, a clean state is provided. This ensures that any potential malware or rogue processes from a previous run don’t carry over.\n    *   Isolation, ephemeral jobs are isolated from the host and other jobs. This can prevent the spread of a potential security breach from one job to another or to the host system.\n\n### 3.5.1.2 Disadvantages\n\n*   Startup overhead, each job incurs an initialization overhead which can be costly for frequent, short-lived tasks.\n*   Resource spikes, concurrent initiation of many single-use jobs can lead to sudden, high demands on resources.\n*   Complexity, workflow tools introduce an additional layer of technology and potential failure points\n\n### 3.5.1.3 Existing workflow tools\n\nFor executing microservice chains as ephemeral jobs in DYNAMOS, Kubernetes jobs are used. Primarily due to their compatibility with Go, which DYNAMOS is written in. Argo does support parallel processing, but then the output of a previous service can not be used.\n\nApache Airflow has to be programmed in Python. Streaming data is a possible solution.\n\n### 4.4.1.1 Streaming messaging\n\nIn server-side streaming, the client sends a request to the server and gets a stream to read a sequence of messages back. Once the client has finished writing the messages, it waits for the server to read them all and return its response. There’s also bidirectional streaming where both sides send messages using a read-write stream. Streaming RPCs are more efficient for large datasets because they break the data up into multiple smaller messages. This however has not been tested yet with regards to passing on SQL data.\n\n## 4.4.2 Microservice chain libraries\n\nTo interface with this gRPC flow, a small library would need to be written for each language that is used in the data exchange microservices. investigating whether we could make an adaptive microservice system that could capture data exchange archetype patterns in microservices. This would decouple date-exchange microservice development from the core data exchange marketplace, allowing for development and deployment without modifying the core system. Furthermore, different data exchange services can be created by composing the microservices in different configurations. Essentially, the system would function akin to a distributed operating system, abstracting away the core microservice complexity so that development can focus on offering data exchange services and the system is able to optimize for extra-functional properties, like energy and optimizing user experience.\n\nIn this chapter, we start by looking at the results from our quantitative experiments described in chapter 5. This will give an initial idea of how this microservice approach behaves and which advantages and disadvantages it brings, which will help define answers to the original research questions.\n\nGiven the size of the DYNAMOS implementation, which encompassed over 8000 lines of code, there were certain research areas that remained unexplored due to time constraints. Consequently, we will conclude this chapter by evaluating potential threats to the validity of our research.\n\n## 7.1 Results discussion\n\nIn this section, we combine the results of the two experiments. We discuss the bottlenecks created by our microservice approach, possible remedies to these bottlenecks and what effect these remedies will have. We start by analyzing the data transfers between services and discuss the cold start time of ephemeral microservice jobs.\n\n### 7.1.1 Data transfers\n\nIn the first experiment, we see the computeToData archetype with three data transfers and the dataThroughTtp archetype with six data transfers.\n\nWith the largest query (30000 lines, 4MB message), around 50% of the time is spent in data transfers in dataThroughTtp versus 23% in the computeToData archetype. Correcting for the startup time of the container, this percentage rises to 62% and 32% respectively. The obvious difference is in the three, respectively six data transfers happening.\n\nIn the alternative implementation of Marten Steketee [36]. The message consists of 4141 lines and is around 8 MB. Two transfers happen, the first from the data provider to the third party by RabbitMQ, which costs 916ms. The second from the third party to the user, 540ms, for a total of 1.46 seconds. The total latency is 2.4 seconds, making data transfers to be also approximately 60% of the processing time.\n\nThis shows two facts. First, encoding and transferring data is done more efficiently in DYNAMOS, this, however, could easily be improved in the other implementation. Second, the number of transfers done in DYNAMOS highly impacts its latency.\n\nDue to the microservice architecture, there will always be more data transfers than in a monolithic application, microservices will need to transfer data to each other. In DYNAMOS, the sidecar pattern, adds to that number of transfers. Completely removing the sidecar will reduce the dataThroughTtp transfers from 6 to 4, and the computeToData archetype from 3 to 2.\n\n&lt;page_number&gt;59&lt;/page_number&gt;\n\n---\n\n\n## Page 61\n\nCHAPTER 7. DISCUSSION\n\nFinding 2: A microservice architecture will increase latency due to extra data transfers, this problem is magnified by the use of a sidecar container\n\n7.1.1.1 Removing the sidecar pattern\n\nRemoving the sidecar pattern by compiling a messaging (RabbitMQ) library into each data-sharing microservices would decrease the amount of data transfers, latency, and the need for the current gRPC data transfer methodology.\n\nHowever, this approach comes with its own set of challenges:\n\n*   **Tight Coupling:** Direct integration would result in a strong dependency on the RabbitMQ platform. Which makes transitioning to another messaging solution more challenging\n*   **Update Overheads:** Any change or update to the messaging library would necessitate modifications, recompilation, and redeployment of all data-sharing microservices\n*   **Maintenance Issues:** Microservices developed in various programming languages would use distinct libraries to connect to a single RabbitMQ instance, potentially leading to increased maintenance complexities\n\nFinding 3: Removing the sidecar pattern leads to a tightly coupled system\n\n7.1.2 Cold starts\n\nA 1.7-second average startup loss is long in the modern web application world. However, in this thesis, we cannot qualify whether this would be unacceptable for a digital data-sharing marketplace solution. One could argue that since the DDM removes a huge amount of effort in getting access to otherwise near-inaccessible datasets, these few extra seconds in latency are trivial.\n\nIf future user requirements however seek to improve this cold start latency, there are ways to mitigate this. The *persistent job* architecture discussed in chapter 3 could be used to avoid cold starts altogether. The trade-off is unnecessary resource consumption and possible security risks (see section 3.5).\n\nFor the *ephemeral job* architecture, an alternative solution would be not to allow the user to add extra options dynamically, like the ‘graph’ option. This would allow the agents to always have a job ready to reduce or avoid cold starts. This comes at the cost of flexibility, user experience, resource consumption, and increased exposure time which is a possible security risk.\n\nFinding 4: Eliminating cold starts results in lowered security standards and could reduce flexibility\n\n7.2 Research questions\n\nThe previous section limited the discussion to the conclusions that could be drawn from the quantitative experiments. In this section, we reflect on the system as a whole by discussing each question in turn.\n\n7.2.1 RQ 1\n\nWhat are the advantages and disadvantages of different approaches to automatically recompose a running microservices architecture in a distributed system?\n\nThroughout the thesis, we discussed the advantages and disadvantages of individual components. In this section, we highlight a few that stand out or are an integral part of DYNAMOS.\n\n7.2.1.1 Microservice chains\n\nThe microservice chain concept explained in section 3.4 and section 4.3 abstract away data-sharing archetypes. The chains solve the question *where* services run and in *which order* the data flows.\n\nThe distributed agents generate the chain the moment a request arrives based on data from the configuration store. This gives DYNAMOS the following options:\n\n---\n\n\n## Page 62\n\nCHAPTER 7. DISCUSSION\n\n* By modifying the configuration store, the orchestrator can dynamically change archetypes without needing the user to be aware\n* Through flags in the request the user can add and remove optional microservices, without involving the orchestrator service\n\n**Finding 5:** The microservice chains are the core of archetype abstraction and data-exchange microservices\n\n### 7.2.1.2 Microservice architectures\n\nWe discussed two microservice architectures, namely an *persistent,-* and *ephemeral* job architecture.\n\n#### Persistent jobs:\n\nThe main advantages of the persistent job architecture are:\n* Faster processing due to lack of cold starts\n* Uses standard messaging for data transfers, avoiding the custom gRPC implementation discussed in section 4.3\n\nThe main disadvantages are:\n* Possibly less secure\n* Constant resource usage\n\n#### Ephemerel jobs\n\nThe main advantages of the ephemeral job architecture are:\n* Complete isolation\n* Resources consumption only when required\n\nThe main disadvantages are:\n* More complexity in managing the jobs\n* Cold starts\n* Possible resource spikes\n\nFor a data steward that only offers a few datasets, with a limited amount of microservices that get queried often, the persistent architecture is a great fit. For another data steward, with more varied, less frequent requests, the ephemeral job architecture could be the better fit.\n\nThe design of DYNAMOS, with distributed agents running on a data steward, allows these choices to be flexible. Whether it be through configuration by the data steward, through policy, or through the agents' decision to switch architectures due to system load, DYNAMOS can select the appropriate architecture and support both, and other scenarios.\n\n**Finding 6:** The distributed agent pattern allows flexible architectures\n\n### 7.2.1.3 Data transfers\n\nWe made a custom data transfer methodology for ephemeral jobs using gRPC. This was to avoid constantly needing to create and remove single-use queues in the messaging platform.\n\n**Advantages:**\n* Data transferred on Inter Process Communication level\n* Language agnostic\n* Loosely coupled\n\n**Disadvantages:**\n* Complex implementation\n* Requires small gRPC library per language used in data exchange microservices\n\n&lt;page_number&gt;61&lt;/page_number&gt;\n\n---\n\n\n## Page 63\n\nCHAPTER 7. DISCUSSION\n\n## 7.2.2 RQ 2\n\n**What attributes/properties should a knowledge base contain to compose microservices in a DDM environment?**\n\nWe solved the challenge of composing microservices in a DDM environment by generating a microservice chain, and abstracting away communication and data transfers, making sure that the data-exchange microservices don’t need to have knowledge of the environment in which they’re orchestrated.\n\nSo to answer this question simply, the info needed for generating a microservice chain is the info required to orchestrate microservices in a DDM environment. Next to a few more practical properties, namely, queue names of microservices (jobs), and endpoints of the distributed agents.\n\nAll of this information, and more, is stored in the configuration management store, etcd, and thus comprises the system state. This system state in etcd, is the core of DYNAMOS’ flexible design. It is a constantly updated source of information for selecting an architecture or archetype, notifies the system of actionable events, and finally, allows developers to configure the system without altering application code.\n\n**Finding 7:** Microservice chains are the core of composing microservices, etcd is the core of making DYNAMOS dynamic and flexible\n\n## 7.2.3 RQ 3\n\n**What extra attributes/properties should a ‘knowledge-base’ contain to make decisions on extra-functional properties, like energy consumption and performance?**\n\nAs discussed in Chapter 3.6, we looked at extra-functional properties such as user experience and energy consumption. We also showed how past a certain point of system load (80%), it can make sense to change archetypes and distribute the load more evenly over the system.\n\nIn Chapter 4.3, we show how DYNAMOS can dynamically adapt archetypes based on a ‘weight’ property. In Chapter 4.5 we demonstrate an architecture on how DYNAMOS can make adaptive choices based on CPU and memory load, although we don’t actually implement the algorithms.\n\nThe answer to this research question ends up not being an exact list of properties. But the assurance, and showcase, of how DYNAMOS can actively adapt to both active and new jobs. The exact algorithms are for a future iteration of DYNAMOS. It is worth noting that it is the responsibility of the distributed agents to expose metrics and threshold information. This means that each data steward can customize how the orchestrator responds to exposed metrics, for example, by setting a lower threshold value.\n\n**Finding 8:** Data stewards, through the distributed agents, can influence how the orchestrator selects archetypes\n\n## 7.3 Further insights\n\nThis design of DYNAMOS provides a clear separation of concerns, creating three main parts of the system. The protocol files define an interface between these components making it explicit what parts of the system can expect from each other.\n\nOne part is the interaction between the orchestrator and the policy enforcer framework. The interface could possibly help the external development of the policy enforcer framework due to the explicitness of the expected results.\n\nThe second part is the combination of the orchestrator and the distributed agents. In which the orchestrator makes archetypes choices, and vice-versa the agents provide information on that system’s state to guide these choices.\n\nAnd finally, the data-exchange microservices are a separate component.\n\n## 7.4 Limitations\n\nThe current implementation of DYNAMOS has a few limitations that we discuss in this section.\n\n&lt;page_number&gt;62&lt;/page_number&gt;\n\n---\n\n\n## Page 64\n\nCHAPTER 7.
