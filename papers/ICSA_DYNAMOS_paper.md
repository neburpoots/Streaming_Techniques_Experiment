DESIGN OF DYNAMOS\n\n&lt;img&gt;Fig. 4. DYNAMOS abstract design&lt;/img&gt;\n\nA data-exchange system is a distributed computing space with a variable number of actors and stakeholders [2]. In this data-exchange system, we introduce two levels of adaptability: first, dynamic composition of microservices to complete different data-exchange requests; second, dynamic archetype selection to achieve extra-functional goals. Figure 4 illustrates the abstract design of DYANMOS. This design involves a distributed agent-based pattern controlled by a central orchestrator, short-lived single-use jobs to facilitate data-exchange services, google protobufs 5, a sidecar pattern, a MAPE-K approach, and an API gateway. The latter three points are discussed in detail below. More details on the decisions taken can be found in the thesis. 6\n\n&lt;img&gt;Fig. 5. Job running on two nodes&lt;/img&gt;\n\n### A. Sidecar pattern\n\nDYNAMOS makes heavy use of the sidecar pattern [7]. Each microservice is deployed with a generic sidecar. The sidecars within DYNAMOS have two functions. First, to abstract communication logic from the exchange layer. In this design a RabbitMQ messaging implementation is used, the sidecar enables swapping this exchange layer for other messaging implementations without affecting the core services.\n\nSecondly the sidecars function as the entry and exit points of a running microservice chain job. Any microservice entering the system will need to comply with this interface. After running a data exchange through the microservice chain, it will end up in the sidecar to route the message to the next node/destination, as seen in Figure 5.\n\n### B. Self-adaptability and MAPE-K\n\nMAPE-K [8, 9] is a common pattern to add self-adaptibility and make decisions about the system based on a *knowledge base*, *etcd* in this case.\n\nWe identify a control loop in an interplay between the distributed agents and the orchestrator. The microservice chain enables dynamic microservice composition, and does this based on the information in the knowledge base. The orchestrator can adapt the jobs in this knowledge base based on information in the legal policies, or other environment input. This allows it to change an archetype *per request*, or dynamically add or remove required microservices.\n\n---\n5https://protobuf.dev/overview/\n6https://github.com/Jorrit05/DYNAMOS/blob/main/docs/papers/Thesis_J_Stutterheim.pdf\n4https://www.openpolicyagent.org/\n\n---\n\n\n## Page 4\n\nC. Platform\n\nEvery node in DYNAMOS runs Kubernetes for executing microservices, chosen for its widespread industry acceptance which makes it a good candidate for running an open-source platform to experiment with microservice chains. Microservice composition - microservice chains\n\nA data exchange microservice chain has a minimum amount of *required* services to process a request. A chain might have *optional* microservices that add extra functionality. Figure 7 shows a valid microservice chain for the SQL data request.\n\nGenerating a chain for this request follows the following steps. All microservices are labeled and stored in the central knowledge base. On receiving a request for data exchange, the process orchestrator checks all policy requirements and creates a ‘composition request’ for each distributed agent, giving each agent a *role* in the process.\n\n```mermaid\ngraph TD\n    subgraph query_service\n        A[query_service]\n    end\n    subgraph anonymize_service\n        B[anonymize_service]\n    end\n    subgraph aggregate_service\n        C[aggregate_service]\n    end\n    subgraph algorithm_service\n        D[algorithm_service]\n    end\n    subgraph graph_service\n        E[graph_service]\n    end\n\n    A -->|optional| D\n    B -->|optional| D\n    C -->|optional| D\n    D -->|optional| E\n```\nFig. 7. On receiving the request the distributed agents check if there is an available composition request, and now can generate a microservice chain using a topological sorting algorithm based on user input, available jobs, and the allocated role in that job.\n\nC. Microservice chain jobs\n\nThe microservice chains are transformed into **single-use, ephemeral jobs**. Ephemeral jobs are created on demand, executed, and subsequently removed. This has advantages in data isolation, and security due to limited exposure and reduced attack surface. However, execution time will be higher due to always needing to spawn new microservices.\n\nD. Job data exchange - gRPC\n\nThe job is created in a single Kubernetes pod (in a Kubernetes job). This pod can contain several microservices to process a data exchange request. We have developed a generic way of using gRPC to pass along data as an argument to a following microservice. This method takes advantage of the following facts:\n\n*   gRPC is language agnostic by default\n*   Data stays in memory, it never writes to disk\n*   A Kubernetes job can be created with multiple containers, inside a single Pod. Pods communicate on ‘localhost’ level, or ‘interprocess communication’ (IPC). This makes the network latency of transferring data from service to service within a Pod negligible\n*   gRPC supports ‘Client streaming RPC’, which means that large datasets can be transferred in a stream of messages\n\n---\n\n\n## Page 5\n\nto preserve internal memory usage. CASE STUDY: UNL**\n\nTo illustrate the practical implications of this research, a proof-of-concept was developed based on the Universities of the Netherlands (UNL) use case from the AMdEX digital data-exchange project [12], focusing specifically on Scenarios 2 and 3 outlined in the AMdEX use cases [2]. This scenario involves a data analyst analyzing SQL datasets, with the same schema, sourced from various universities across the Netherlands.\n\nThe scenario necessitates meeting several key requirements. Firstly, it should enable the selection of a different archetype for each request, accommodating changes in contract agreements, user requests for data aggregation requiring third-party involvement, or shifts in data processing to third parties due to nearing capacity at data nodes. DYNAMOS is expected to autonomously determine the appropriate archetype for each request. Additionally, all decisions made must be subject to validation by a policy evaluator. Moreover, the system must ensure programming language agnosticism for broader compatibility and flexibility.\n\nThe data analyst sends out an SQL query and an algorithm, on which DYNAMOS checks the policy, generating a microservice chain, and lets the universities deploy the services, returning the results without requiring the data analyst to interact with any data. This scenario is executed using two archetypes, namely ‘ComputeToData’ and ‘Data via Trusted third party (DataThroughTtp)’ [3], seen in Figure 8.\n\nTo realize the intended objectives of this scenario, five microservices were specifically created in addition to the services discussed in section IV. These services include a Query service, an Anonymize service dedicated to anonymizing sensitive data, an Aggregate service consolidating data from multiple queries, an Algorithm service executing data analysis according to specified algorithms, and ultimately, a Graph service that optionally transforms results into graphical representations.\n\nThe distributed architecture to create this PoC is depicted in Figure 6. It recognizes a **control plane** and **data plane**. In the control plane, the process orchestrator makes archetypes, and job decisions in conformance with a policy evaluator. In the data plane, distributed agents are responsible for scheduling the services, and updating the current system state in the *distributed datastore*.\n\n**VI. Architecture - lessons learned\n\nSeveral key observations emerged during this project. In this section, we delve into the primary findings relating to both design and implementation perspectives.\n\n1) **Sidecar pattern**: The Sidecar pattern is heavily utilized in DYNAMOS, as explained in subsection III-A. During development, we considered removing the sidecar from the architecture by compiling a messaging (RabbitMQ) library into each data-sharing microservices. This decision was based on the fact that the sidecar pattern would increase the amount of data transfers, latency, and complexity. Removing the sidecar would remove the need for extra gRPC data transfers, in turn simplifying the overall implementation.\n\nHowever, the sidecar pattern does provide some advantages, notably relating to the separation of concerns. Different dataspaces could theoretically use different exchange systems (RabbitMQ/Kafka/HTTP). By swapping sidecars to connect with core components, we isolate the communication functionality, enhancing modularity and flexibility.\n\n2) **Protocol buffers**: Another observation relates to the utilization of Google’s Protocol Buffer messages. The integration of protocol buffers offered a standardized and language-independent method for facilitating communication between services. Although this proved pivotal in the design and implementation of DYNAMOS, it heightens the complexity of the codebase. This complexity originates from requiring different gRPC implementations per language. Although the interface is language agnostic, the implementation libraries are language-specific.\n\nC. Ephemeral jobs\n\nEphemeral jobs, which operate on transient data without storing it, minimize exposure and decrease the attack surface, enhancing security. Their single-use nature ensures a clean state after execution.\n\nHowever, this approach introduces complexity and leads to latency, stemming from the need for precise timing coordination among services for message exchange.\n\n1) **Cold starts**: A 1.7-second average startup loss is long in the modern web application world. However, in this research, we cannot qualify whether this would be unacceptable for a digital data-exchange marketplace solution. One could argue that since the dataspace removes a large amount of effort in getting access to otherwise near-inaccessible datasets, these few extra seconds in latency are trivial.\n\nIf future user requirements however seek to improve this cold start latency, there are ways to mitigate this. Using a *persistent job* (the counter-part to ephemeral jobs) architecture could be used to avoid cold starts altogether. The trade-off is unnecessary resource consumption and possible security risks.\n\nFor the *ephemeral job* architecture, an alternative solution would be not to allow the user to add extra options dynamically, like the ‘graph’ option (see section V). This would allow the agents to always have a job ready to reduce or avoid cold starts. This comes at the cost of flexibility, user experience, resource consumption, and increased exposure time which is a possible security risk.\n\n2) **Data transfers**: We devised a customized data transfer approach for ephemeral jobs using gRPC to circumvent the frequent creation and removal of single-use queues in the messaging platform. This methodology offered several advantages: data was transferred at the Inter-Process Communication level, ensuring language agnosticism and loose coupling. However, it entailed some drawbacks, including a complex implementation process and the necessity for a small gRPC library per language utilized in the data exchange microservices.\n\nD. Further insights\n\nThis design of DYNAMOS provides a clear separation of concerns, creating three main parts of the system. The protocol files define an interface between these components making it explicit what parts of the system can expect from each other.\n\nOne part is the interaction between the orchestrator and the policy enforcer framework. The interface could help the external development of the policy enforcer framework due to the explicitness of the expected results.\n\nThe second part is the combination of the orchestrator and the distributed agents, in which the orchestrator chooses archetypes; vice-versa, the agents provide information on that system’s state to guide these choices.\n\nThirdly, the data-exchange microservices are a separate component.\n\nE. Threats to validity\n\n1) **Specific sample use case**: In this research, we took two archetypes and the UNL use case as the basis for the DYNAMOS implementation. We believe the implementation of the language to generate microservice chains, including the way archetypes are defined, is general enough to extend to more use cases and archetypes. This, however, is not proven and requires a more exact analysis of different use cases, request types, and archetypes.\n\n2) **Generic microservices**: We made a generic method of passing data from microservice to microservice and showed with a small example how this data can be manipulated. However, we fell short of realizing the full chain as described in the use case. With our current implementation, we cannot prove that this method of passing data is user-friendly, or generic enough to scale up toward more complex data-sharing scenarios.\n\n3) **Authorization and data pods**: The generic architecture hinges on the idea that we can implement web tokens, that can be invalidated, to access data pods on data stewards. This has not been tested, or researched much for DYNAMOS, although we believe that the techniques and technology do exist to make this happen.\n\n4) **Message size**: Streaming data would be a possible method of working with larger datasets. How this would be exactly processed by microservices expecting, for example, SQL data, has not been properly researched in this paper.\n\n---\n\n\n## Page 7\n\n5) **HTTP**: For communication between the microservices, whether it be for persistent or ephemeral jobs, we solely focused on an AMQ or gRPC approach in this thesis. We believe in this scenario it makes sense to have stable queueing in the form of AMQ, and in ephemeral jobs to have the direct IPC communication of gRPC. However, we never fully investigated whether HTTP should be completely discarded as an option. We believe putting some extra effort into listing out the advantages and disadvantages of using HTTP or other methods would be beneficial.\n\n**F. Future work**\n\nThere are several research directions in which we would like to take this.\n\n1) **Enhance Compliance**: A logical next step would be to implement a data pod solution (similar to The Solid Project <sup>8</sup>) and show how a user can get authentication to a dataset in that specific data pod, and nowhere else. This entire flow could be essential in the dataspace, irrespective of how the final application is designed. Additional security features relating to auditability and privacy would also enhance compliance. For example, utilizing a checksum for each message that is audited and validated by an auditor service.\n\n2) **Archetype expansion**: To see if the current microservice chain generation holds, it is essential to expand our research domain by incorporating additional use cases and archetypes. An interesting use case would be the KLM federated machine learning use-case for predictive airline maintenance<sup>9</sup>. Furthermore, we would like to improve the algorithms for the dynamic selection of archetypes. Especially algorithms concerning ‘green-IT’[13] and resource consumption require more extensive research.\n\n3) **Industry scale use case**: To validate our concept at an industry scale, we plan to deploy our use cases on a scientific testbed, such as Fabric<sup>10</sup>. Furthermore Fabric would enable to also investigate whether control plane components (orchestration, policy) can be moved into the network infrastructure, as that would enable the network to act as a trusted third party running the orchestration. To evaluate DYNAMOS at an industry scale, we start collaborating with KLM and Equinix on the predictive airline maintenance use case.\n\n**VII. Self-adaptive microservice systems**\n\nResearch on self-adaptive microservices tends to be focused on architecture and modeling [8, 14–16]. Mendonça et al. acknowledge the limited scope of current works addressing the challenges of developing self-adaptive microservice applications [17–19]. They strive to bridge the gap between state-of-the-art research and practice, particularly emphasizing the future of self-adaptive microservices. Their work proposes a microservices service-mesh as the path to self-adaptability, exemplified by KubowMesh, a modern microservice approach built on Kubernetes and the Istio service mesh [17, 18] based on the self-adaptability patterns of Rainbow [20].\n\nWhile Mendonça et al. provide a practical approach, other papers contribute key ideas. The concept of distributed agents orchestrating microservices in a heterogeneous system draws from the computing-fleet reference architecture, which proposes a three-layered architecture for microservice orchestration on edge computing devices [14]. Many papers on self-adaptive (microservice) systems predominantly focus on ‘Quality of Service’ attributes such as costs, seamless upgrades, resiliency, and scaling. For instance, a comprehensive literature review on QoS-driven service composition under uncertainty provides a taxonomy, comparisons, and analysis of state-of-the-art approaches [15]. Additionally, a multi-level self-adaptation architecture and domain-specific language (DSL) proposal addresses seamless service upgrades [16].\n\nThe work of Arcaini et al. [8] formalizes the modeling of MAPE-K systems in distributed systems, discussing aspects of validation and correctness [8]. Boyapati et al. [9] implement a lightweight MAPE-K loop to control dynamic rate limiting in a Docker compose setup.\n\n**VIII. CONCLUSION**\n\nWitness to the growing public and private sector’s interest in data exchange, recent literature introduces several approaches to how DDM architecture could be created.\n\nDYNAMOS makes this practical. It can be used to experiment with security, different archetypes, and how to interact with a policy enforcer framework. It allows for each part of the system to operate distinctly, just like an actual data steward might want to do things differently from another.\n\nWe defined microservice chains as a directed acyclic graph, created a custom method of passing data between services in short-lived ephemeral jobs, designed two microservice architectures, and incorporated two data-sharing archetypes.
